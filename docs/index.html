1. What Is This Robot?

This is a small robot with 4 wheels that can move in any direction — forward, backward, sideways, diagonally, or spin in place. It's controlled by sending text commands over a USB cable from your computer (or wirelessly with a PS2 gamepad).

Think of it like this: Imagine a remote-controlled car, but instead of only going forward and turning, it can also slide sideways like a crab. This is possible because of special wheels called Mecanum wheels that have small rollers at 45-degree angles.

What Can It Do?

↺

Move In Any Direction

10 movement directions: forward, backward, left, right, 4 diagonals, rotate clockwise, and rotate counter-clockwise.

🎯

Move Exact Distances

Tell it "go forward 10 cm" and it measures its wheel rotations to stop at exactly the right spot. Each wheel has a sensor (encoder) that counts 4,320 ticks per revolution.

⚡

Self-Correcting Speed

A built-in speed controller (PID) checks each wheel's speed 50 times per second and adjusts power automatically. If one wheel is slower, it gets more power.

💻

Computer or Gamepad Control

Send text commands from your PC ("FWD,100,2000"), use a Python GUI with keyboard controls, or plug in a PS2 wireless gamepad.

2. How It All Works Together

Here is the journey of a command from your computer all the way to the wheels spinning. Understanding this flow makes everything else in this document make sense.

The Full Journey of a Command

Let's say you type FWD,100,2000 (meaning "go forward at speed 100 for 2000 encoder ticks"). Here's what happens step by step:

1

You Send a Command

Your PC sends the text FWD,100,2000\n over USB at 115,200 baud (characters per second). The Arduino receives it one character at a time.

2

The Command Parser Reads It

serial_cmd.cpp collects characters until it sees a newline (\n), then splits the text by commas to understand: command = FWD, speed = 100, ticks = 2000.

3

The State Machine Accepts It

robot.cpp checks: "Am I idle? Yes." So it accepts the command, resets the encoder counters to zero, and switches from IDLE state to MOVING state. It sends back OK.

4

Mecanum Math Calculates Wheel Directions

mecanum.cpp figures out that "forward" means all 4 wheels spin forward at the same speed. (For "strafe left" it would be a different combination — see the Mecanum section.)

5

The Speed Controller (PID) Runs 50 Times Per Second

Every 20 milliseconds, motion.cpp checks each wheel's encoder to see how fast it's actually spinning, compares that to how fast it should be spinning, and adjusts the motor power up or down. This is like cruise control in a car — if you hit a hill, it gives more gas.

6

Motor Driver Sends Power

motor.cpp sends the calculated power level to the PCA9685 chip over I2C. This chip generates PWM signals (rapid on/off pulses) that control how much power each motor gets.

7

H-Bridges Deliver Power to Motors

The TB67H450 H-bridge chips receive the PWM and switch battery power to the motors. Two signals per motor (IN1/IN2) control direction: one high = forward, the other high = backward.

8

Target Reached → Stop

When the average encoder count reaches 2000 ticks, the robot brakes briefly (40ms) to stop quickly, then releases the motors (coast mode = no power), and sends DONE back to your PC.

Key insight: The Arduino does NOT just "turn the motor on and wait." It checks and adjusts each wheel's speed 50 times per second using encoder feedback. This closed-loop control is what makes the robot move straight and stop accurately.

4. The Brain — Robot States

The Arduino runs a state machine — at any moment, the robot is in one of 5 "states" (like modes). What commands it accepts and what it does depends on which state it's in.

What's a state machine? Think of a traffic light: it's either RED, YELLOW, or GREEN. It follows rules about when to switch (RED → GREEN → YELLOW → RED). Our robot is similar — it's either IDLE, MOVING, CALIBRATING, TESTING, or ERROR, and specific events cause it to switch between these modes.

Safety Features

The robot has several automatic protections to prevent damage:

Stall Detection

Problem: A wheel is stuck (encoder not moving).
Response: If any motor makes less than 50 ticks progress in 5 seconds, the robot stops and reports "Stalled".

Movement Timeout

Problem: A move is taking way too long (something is wrong).
Response: Any single movement automatically stops after 30 seconds.

Watchdog Timer

Problem: In velocity mode, the PC crashed or disconnected.
Response: If no VEL command arrives within 200ms, the robot stops. The PC must keep sending commands to keep moving.

I2C Health Check

Problem: The wire to the PCA9685 came loose.
Response: Checked every 2 seconds. After 5 failures, enters ERROR state and turns off all motors.

5. Motors & Wiring

The robot has 4 DC motors, each connected to an H-bridge driver and an encoder sensor. This section explains which motor is connected to which chip pins.

What's an H-bridge? A motor spins one way when current flows through it in one direction, and the opposite way when current flows in reverse. An H-bridge is a circuit that can switch the current direction electronically. Each of our TB67H450 H-bridges has two control pins:
  • IN1 gets power (IN2 off) → motor spins forward
  • IN2 gets power (IN1 off) → motor spins backward
  • Both on → electric brake (motor stops quickly)
  • Both off → coast (motor freewheels, no current)

6. Speed Control (PID)

If you just set a motor to a fixed power level, it might spin faster or slower depending on battery charge, floor friction, or load. The PID controller automatically adjusts power to maintain the exact speed you want.

Analogy: Cruise control in a car. You set the speed to 60 km/h. Going uphill, the car gives more gas. Going downhill, it gives less. It constantly checks the speedometer and adjusts. Our PID does the same thing, but for each wheel, 50 times per second.

How the PID Controller Works

Each of the 4 motors has its own independent speed controller. Every 20ms (50 times per second), it does this:

1

Read the encoder

Count how many ticks the wheel moved since the last check (20ms ago). This is the actual speed (ticks per 20ms period).

2

Calculate the error

error = desired speed - actual speed
If error is positive, the wheel is too slow. If negative, too fast.

3

Calculate Feed-Forward (initial guess)

Based on calibration data, estimate how much power (PWM) is needed for this speed.
FF = desired_ticks_per_period × 255 / max_ticks_per_period_for_this_motor
This gets us close, but not perfect (that's what the next step fixes).

4

Calculate PI Correction

P (Proportional): 1.5 × error — big correction if the error is big.
I (Integral): 0.3 × accumulated_error — slowly builds up if there's a persistent small error (like a slightly weak motor). Capped at ±150 to prevent runaway.

5

Set the motor power

PWM = Feed-Forward + P correction + I correction
Clamped to 0-255, then sent to the PCA9685 chip.

PID Parameters (in config.h)

ParameterValueWhat It Means
Kp (Proportional)1.5 For every 1 tick/period the motor is too slow, add 1.5 PWM. Higher = more aggressive correction.
Ki (Integral)0.3 Slowly accumulates correction for persistent errors. Lower = less overshoot but slower to respond.
IMAX (integral limit)150 Maximum value the integral can reach. Prevents "windup" (overcorrection after a stall).
Control frequency50 Hz PID runs every 20 milliseconds. Fast enough for smooth control, slow enough for the Arduino to handle.

Speed Ramping (Smooth Start & Stop)

Why not just go full speed immediately? If you suddenly apply full power, the wheels can slip on the floor (like flooring the gas pedal). And if you stop instantly, the robot overshoots its target. So we ramp the speed up at the start and down near the end.
PhaseWhenWhat HappensParameters
Acceleration First 300 ticks (~1.7cm) Speed ramps up linearly from 40 to the target speed. Prevents wheel slip on startup. ACCEL_RAMP_TICKS=300, ACCEL_MIN_SPEED=40

7. Mecanum Wheels — How Omnidirectional Movement Works

Normal wheels can only go forward and backward. Mecanum wheels have angled rollers that create sideways forces. By spinning different wheels in different directions, the robot can move any way you want.

How it works: Each Mecanum wheel has small rollers at 45 degrees. When the wheel spins, it pushes both forward AND sideways. By choosing which wheels go forward and which go backward, you can combine these forces to move in any direction.

Velocity Mode (VEL Command) — Inverse Kinematics

Instead of picking a named direction, you can directly specify velocity components:

  • vx = forward/backward speed (-255 to 255)
  • vy = left/right speed (-255 to 255, positive = left)
  • wz = rotation speed (-255 to 255, positive = counter-clockwise)

The firmware converts these into per-motor speeds using this formula:

FL = vx - vy - wz
FR = vx + vy + wz
RL = vx + vy - wz
RR = vx - vy + wz

// If any value exceeds 255, all are scaled down proportionally
Example: VEL,100,50,0 means "go forward at speed 100 while also strafing left at speed 50." The math combines these into the right wheel speeds for a diagonal movement.

8. Serial Command Reference

You control the robot by sending text commands over USB Serial. Each command is a line of text ending with a newline character. The format is: COMMAND,param1,param2\n

Connection settings: 115200 baud, 8 data bits, no parity, 1 stop bit (8N1). You can use the Arduino Serial Monitor, minicom, screen /dev/ttyACM0 115200, or the included Python GUI.

9. Encoders — How the Robot Knows Where It Is

Each wheel has a sensor called a quadrature encoder that counts tiny steps as the wheel rotates. This is how the robot measures distance traveled and wheel speed.

Analogy: Imagine painting 4,320 tiny lines around the inside of a wheel. A sensor counts how many lines pass by as the wheel spins. One full wheel rotation = 4,320 counts ("ticks"). Since the wheel circumference is 251.3mm, each tick represents 0.058mm — about the width of a hair.

10. Motor Calibration

No two motors are exactly the same. One might spin slightly faster than another at the same power level. Calibration measures each motor's actual top speed so the PID controller can compensate.

Why calibrate? The PID controller uses a "feed-forward" estimate: "to go at speed X, I probably need about Y power." If the robot doesn't know each motor's characteristics, this estimate will be wrong, and the PID has to work harder to correct. After calibration, the estimate is accurate and the wheels run much more smoothly.

What Happens During Calibration

1

Send the CALIB command

The robot enters CALIBRATING state. All other commands are rejected (except STOP).

2

Session 1 of 3: Motors spin up

All 4 motors run forward at a fixed power level (PWM=200 out of 255). Wait 500ms for them to reach steady speed.

3

Session 1 of 3: Measure speed

For 2 seconds, the firmware counts how many encoder ticks each motor produces. This gives the actual tick rate for each motor at PWM=200.

4

Brief pause

Motors brake for 300ms, then repeat. Three sessions total for a more accurate average.

5

Calculate & save

The average tick rate at PWM=200 is extrapolated to PWM=255 (theoretical max speed). These values (one per motor) are saved to the Arduino's EEPROM (permanent memory that survives power off).

The 0xCB marker byte tells the firmware "yes, there's valid calibration data here." On boot, if this marker is found, the calibration data is loaded. If not, a default value of 150 ticks/period is used for all motors. Typical calibrated values are 148-162 depending on the individual motor.

11. Hardware Test Firmware

A separate, simpler firmware for testing individual motors and encoders. No PID, no state machine — just direct control. Useful for verifying that wires are connected correctly.

How to Use It

# Build the test firmware (does NOT overwrite the main firmware in memory)
pio run -e hwtest

# Upload it to the Arduino (replaces the main firmware temporarily)
pio run -e hwtest -t upload

# Open serial monitor to send commands
pio device monitor -e hwtest

# To go back to normal firmware later:
pio run -e uno -t upload
Note: The regular pio run (without -e hwtest) only builds the main firmware. You must explicitly specify -e hwtest to build or upload the test version.

Quick Test — Verify Everything Works

Follow these steps after uploading the firmware to confirm the robot is working:

1

Check communication

Open serial monitor and send READ. You should see 4 encoder values (all 0 if the wheels haven't moved). If you see READY on startup, the Arduino is working.

2

Test one motor

Send TMOTOR,FL,100. The front-left wheel should spin. The FL encoder should count up. The other 3 encoders should stay near 0. If a different wheel spins, your wiring is swapped!

3

Test forward movement

Send STOP first, then FWD,50,500. The robot should move forward a short distance and then stop. You should receive OK then DONE.

4

Check encoder readings

Send READ again. All 4 encoder values should be roughly equal (within ~10% of each other). If one is very different, that motor/encoder might have a wiring issue.

5

Run calibration

Send CALIB. The robot will drive forward 3 times (~10 seconds total). When it finishes, you'll see the calibrated values printed and DONE. These values are saved and will be loaded automatically on next power-on.
